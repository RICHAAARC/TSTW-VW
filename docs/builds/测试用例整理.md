# 测试用例整理方案

## 一、文档定位

本文档用于指导 Codex 对当前项目测试用例进行结构化整理。整理目标不是删除覆盖面，而是降低测试文件碎片化、消除单一文件内不同粒度测试混杂、明确默认测试口径，并使测试目录结构与项目文件组织约束一致。

当前 `docs/test_list.md` 显示：项目共有 226 个测试节点，当前默认 `pytest -q` 口径执行 136 个节点，排除 90 个节点。当前默认口径并非纯轻量测试，仍包含部分 mp4 读写、backend 构造、packager tiny run 和真实视频 scaffold 测试。因此，本次整理应以“验证内容”和“典型耗时”为主线，将测试分为约束测试、功能测试和集成测试三层。

本文档面向当前仓库状态制定，目标目录为：

```text
tests/
├── constraints/
│   ├── test_naming.py
│   ├── test_boundary.py
│   ├── test_protocol_contracts.py
│   └── ...
├── functional/
│   ├── test_mp4_io.py
│   ├── test_vae_backend.py
│   ├── test_real_video_attacks.py
│   └── ...
├── integration/
│   ├── test_real_video_probe_smoke.py
│   ├── test_real_video_metrics_smoke.py
│   ├── test_formal_gate.py
│   └── ...
├── fixtures/
├── helpers/
└── conftest.py
```

本次整理必须保持以下基本原则：

```text
1. 不降低现有测试覆盖面；
2. 不通过删除测试来缩短默认耗时；
3. 不改变被测业务逻辑；
4. 一个测试文件只服务一种主要测试类型；
5. 约束测试默认全执行；
6. 功能测试默认只执行 quick 子集；
7. integration / smoke / slow / formal 默认排除，由 CI 分层调度；
8. 重量级资源使用 session 级 fixture 缓存，禁止同一测试文件内反复生成 tiny run root；
9. 测试文件命名必须表达测试内容，不使用 stage2、p2、v1 等弱命名；
10. 测试目录调整后必须同步更新 `docs/test_list.md`、`pyproject.toml` 与必要的审计脚本。
```

## 二、当前主要问题

### （一）目录平铺导致测试类型不可见

当前所有测试基本平铺在 `tests/` 根目录。仅从文件路径无法判断测试属于：

```text
纯静态约束；
轻量功能；
真实 I/O；
packager；
runner smoke；
formal gate；
真实视频攻击；
真实视频质量指标；
历史 synthetic probe。
```

这会导致默认 `pytest -q` 难以稳定控制耗时，也不利于 CI 分层。

### （二）marker 策略过粗

当前 `pyproject.toml` 使用：

```toml
addopts = ["-m", "not slow and not formal and not smoke"]
markers = [
    "unit: fast unit-level tests",
    "smoke: lightweight runtime smoke tests",
    "slow: longer proof-profile or broad integration tests",
    "formal: formal-profile completion evidence tests",
]
```

问题是：

```text
1. `unit` 没有成为默认选择的正向条件；
2. `constraint` 没有作为治理类测试别名；
3. `quick` 没有表达轻量功能测试；
4. `integration` 没有区分真实 I/O / runner / packager 类测试；
5. 仅靠 not slow / not formal / not smoke 无法排除未标记但偏重的功能测试；
6. 当前有些功能测试默认执行，但实际会写 mp4、构建 backend 或运行 tiny runner。
```

### （三）部分文件混合不同粒度

`docs/test_list.md` 已标出典型混合文件：

```text
tests/test_real_video_ffmpeg_compression_attack.py
  同时包含轻量 guard / contract 与真实 ffmpeg compression smoke。

tests/test_real_video_tar_zst_packager.py
  同时包含纯 packager contract 与完整 tiny stage-two run archive 检查。

tests/test_records_schema.py
  同时包含轻量 schema 断言与依赖 smoke runtime 的 records 闭环。
```

这类文件应拆分为：

```text
test_xxx_constraints.py
  保留纯静态、轻量 contract、命名、schema、guard 测试。

test_xxx_functional.py
  保留真实 I/O、backend 构造、函数行为测试。

test_xxx_smoke.py
  保留 runner、packager、artifact rebuild、真实 ffmpeg 路径等集成测试。
```

### （四）多个小而同质的约束测试过于碎片化

当前存在多个节点数较少且同质的治理测试，例如：

```text
test_core_boundary_layering.py
test_repository_intake.py
test_stage_boundaries.py
test_formal_rejects_placeholder_vae.py
test_real_video_vae_latent_backend_no_synthetic_dependency_formal.py
test_real_video_vae_latent_no_flow_matching_dependency.py
test_synthetic_tubelet_sync_no_dit_dependency.py
```

这些测试均属于轻量边界 / dependency gate / stage boundary 约束，适合按主题合并，减少文件碎片。

### （五）重量级资源没有统一 session 级缓存

当前部分测试通过 helper 反复生成 tiny run root 或 mp4 artifact。例如：

```text
test_real_video_tar_zst_packager.py
test_real_video_vae_latent_table_rebuild.py
test_records_schema.py
```

这类测试应统一通过 `tests/conftest.py` 或 `tests/helpers/` 提供 session 级 fixture：

```text
session_tiny_mp4_dataset
session_real_video_probe_tiny_run
session_mock_vae_model_root
session_ffmpeg_available
session_lpips_fixture
```

禁止同类测试文件内部重复构建相同 heavy fixture。

## 三、目标 marker 策略

### （一）marker 定义

将 `pyproject.toml` 更新为：

```toml
[tool.pytest.ini_options]
testpaths = ["tests"]
pythonpath = ["."]
addopts = ["-m", "(constraint or unit or quick) and not integration and not smoke and not slow and not formal"]
markers = [
    "unit: extremely fast tests with no I/O and no external dependency",
    "constraint: static or lightweight governance tests; treated as unit-level by default",
    "quick: lightweight functional tests that may use tiny in-memory fixtures but should finish quickly",
    "integration: tests requiring real I/O, backend construction, runner instantiation, or artifact directories",
    "smoke: key end-to-end path tests for CI smoke pipeline; excluded from local default unless explicitly selected",
    "slow: tests that may exceed 30 seconds or perform broad sweeps",
    "formal: tests dedicated to formal gate or formal-profile completion evidence",
]
```

需要注意：`constraint` 与 `unit` 可以共存。建议治理类测试统一加：

```python
@pytest.mark.constraint
@pytest.mark.unit
```

纯函数功能测试统一加：

```python
@pytest.mark.unit
```

轻量功能测试统一加：

```python
@pytest.mark.quick
```

真实 I/O、runner、packager、artifact rebuild、真实 ffmpeg、真实 VAE 路径统一加：

```python
@pytest.mark.integration
```

关键端到端 smoke 测试加：

```python
@pytest.mark.integration
@pytest.mark.smoke
```

formal gate 测试加：

```python
@pytest.mark.formal
```

若 formal gate 本身需要 runner 或 I/O，则加：

```python
@pytest.mark.integration
@pytest.mark.formal
```

### （二）推荐执行口径

本地默认快速检查：

```bash
python -m pytest -q
```

等价于：

```bash
python -m pytest -q -m "(constraint or unit or quick) and not integration and not smoke and not slow and not formal"
```

只跑约束测试：

```bash
python -m pytest -q tests/constraints -m "constraint or unit"
```

只跑轻量功能测试：

```bash
python -m pytest -q tests/functional -m "quick or unit"
```

CI 主流水线建议：

```bash
python -m pytest -q -m "(constraint or unit or quick or smoke) and not slow and not formal"
```

重型 integration 检查：

```bash
python -m pytest -q tests/integration -m "integration"
```

formal gate 检查：

```bash
python -m pytest -q -m "formal"
```

全量检查：

```bash
python -m pytest -q -m "constraint or unit or quick or integration or smoke or slow or formal"
```

用于纯静态、治理、命名、边界、schema、contract、dependency gate、formal rule 的轻量测试。原则上不得执行真实 mp4 写入、ffmpeg、runner、packager 或 VAE encode。

### （二）`tests/functional/`

用于验证具体函数、backend registry、视频 IO、攻击函数、指标函数、packager 函数等轻量功能。可以使用 tiny fixture，但默认只允许 `unit` 或 `quick` 子集执行。

推荐文件：

```text

```

### （三）`tests/integration/`

用于真正重量级的 smoke、runner、artifact rebuild、formal gate、真实 ffmpeg、真实视频指标、完整 attack matrix 和 stage-specific pipeline。默认本地排除。

推荐文件：

```text

tests/integration/test_real_video_attack_matrix.py
tests/integration/test_real_video_metrics_smoke.py

```

### （四）`tests/helpers/` 与 `tests/fixtures/`

推荐结构：

```text

tests/helpers/real_video_probe.py
tests/helpers/mp4_fixtures.py
tests/helpers/mock_vae.py
tests/helpers/run_roots.py

├── tiny_video_frames.json
├── tiny_manifest.json
└── README.md
```

公共 helper 不应命名为 `test_*.py`，避免被 pytest 收集为测试文件。
