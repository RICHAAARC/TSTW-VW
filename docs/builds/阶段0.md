## 一、阶段 0 的定位

阶段 0 应定义为 **protocol skeleton and record freeze**，即“协议骨架与记录层冻结”。其核心目标不是验证水印有效性，而是先冻结未来所有阶段必须遵守的实验协议、记录格式、阈值生成路径、消融入口和产物重建路径。项目构建文件明确要求阶段 0 固定 `sample role`、`split`、`record schema`、`threshold protocol`、`method variant`、`evidence interface` 与输出结构，并强调该阶段“不追求水印性能，只追求协议可执行、记录可审计、结果可复现”。方法机制文件也明确规定，阶段 0 的目标是建立代码骨架，并固定 `sample role`、`split`、`record schema`、`threshold protocol` 与 `evidence interface`。

因此，阶段 0 的最小结论应为：

```text
本阶段只证明：
dev / calibration / test 协议可执行；
threshold 只由 calibration negative 产生；
event-level records 可完整写出；
tables 可由 records 自动重建；
placeholder / random 字段可被显式识别并在后续阶段清理或替换。
```

阶段 0 不应证明：

```text
tubelet_sync 的真实鲁棒性；
trajectory evidence 的独立增益；
DiT / Flow Matching 采样过程约束有效；
真实视频质量或时序一致性可控。
```

## 二、阶段 0 的命名总原则

你提出的命名约束是合理的。阶段 0 不应大量使用 `_v1`、`_p0`、`stage0`、`probe0` 这类语义弱且后期难以清理的标识。建议采用如下规则。

### （一）正式字段命名规则

正式协议字段应使用稳定语义命名，例如：

```text
protocol_name
construction_phase
method_family
method_variant
evidence_family
score_name
threshold_id
calibration_policy
split_policy
record_schema_name
manifest_schema_name
```

其中，`construction_phase` 推荐取值为：

```text
protocol_skeleton
synthetic_latent_probe
real_video_latent_probe
trajectory_statistic_probe
trajectory_aware_embedding
full_paper_protocol
minimal_release
```

不建议使用：

```text
stage_0
stage_1
protocol_v1
probe_p0
method_v1
```

如果已有外部配置或历史 schema 必须保留 `project_stage` 字段，则建议增加一个更稳定的并行字段：

```json
{
  "project_stage": "stage_0",
  "construction_phase": "protocol_skeleton"
}
```

这样既兼容既有 schema，又避免后续论文口径依赖阶段编号。

### （二）占位字段命名规则

凡是后续必须替换、当前只是接口占位的字段，必须显式使用 `_placeholder` 后缀。例如：

```text
watermark_payload_placeholder
trajectory_observation_placeholder
sync_alignment_placeholder
quality_score_placeholder
motion_consistency_placeholder
semantic_consistency_placeholder
sampling_constraint_placeholder
```

这些字段的含义是：当前没有真实实现，只是为了冻结 schema 和接口。

### （三）随机字段命名规则

凡是当前由伪随机数、模拟分数或 synthetic dummy 数据生成的字段，必须显式使用 `_random` 后缀。例如：

```text
latent_tensor_digest_random
tubelet_score_random
sync_score_random
trajectory_score_random
final_score_random
score_generation_seed_random
random_score_distribution_id
```

这些字段的含义是：当前值不是方法性能证据，只用于验证 pipeline、records、threshold 与 table builder 可运行。

### （四）禁止命名

阶段 0 中禁止新增以下类型命名：

```text
score_v1
method_p0
detector_stage0
threshold_probe_v1
event_schema_v1
table_p0
```

如果确实需要 schema 版本，应使用明确的 schema 字段，而不是混入业务字段名：

```json
{
  "record_schema_name": "event_score_record",
  "record_schema_semver": "0.1.0"
}
```

## 四、阶段 0 的构建顺序

### （一）第一步：冻结枚举与 schema

首先实现 `main/core/schema.py`，只做三件事：

1. 固定 split 枚举；
2. 固定 sample role 枚举；
3. 固定 event score record、threshold record、run manifest 的必填字段。

推荐枚举如下：

```python
SPLITS = {
    "dev",
    "calibration",
    "test",
}

SAMPLE_ROLES = {
    "clean_negative",
    "attacked_negative",
    "watermarked_positive",
    "attacked_positive",
}

EVIDENCE_SCORE_NAMES = {
    "S_tubelet",
    "S_sync",
    "S_traj",
    "S_final",
}
```

注意：`S_tubelet`、`S_sync`、`S_traj` 在阶段 0 可以为 `null` 或由 random 占位产生，但字段必须存在。项目构建文件明确要求每条 score record 都包含 split、sample role、method variant、attack name、evidence scores、final score 与 decision。

---

### （二）第二步：冻结配置入口

阶段 0 所有实验差异必须由配置控制，不允许 notebook 临时覆盖或函数内部硬编码。项目构建路线明确禁止手动注释代码、修改函数内部常量、复制脚本、notebook 临时覆盖关键协议参数且不写入 records、基于 test positive 反向调阈值等行为。

推荐配置如下：

```yaml
# configs/protocol/fixed_low_fpr.yaml
protocol_name: fixed_low_fpr_calibrated_detection
construction_phase: protocol_skeleton
target_fpr: 0.001
score_name: S_final
calibration_split: calibration
calibration_negative_roles:
  - clean_negative
  - attacked_negative
threshold_quantile_rule: upper_tail
allow_test_threshold_update: false
allow_attack_specific_threshold: false
```

```yaml
# configs/method/random_score_detector_random.yaml
method_family: temporal_tubelet_watermark
method_variant: random_score_detector_random
method_status: random
enabled_evidence:
  tubelet: true
  sync: true
  trajectory: true
fusion_rule: random_score_linear_fusion_random
score_generation_seed_random: 20260506
```

```yaml
# configs/method/empty_watermark_method_placeholder.yaml
method_family: temporal_tubelet_watermark
method_variant: empty_watermark_method_placeholder
method_status: placeholder
enabled_evidence:
  tubelet: false
  sync: false
  trajectory: false
fusion_rule: constant_zero_fusion_placeholder
```

这里需要同时保留 `empty_watermark_method_placeholder` 与 `random_score_detector_random`。前者用于测试禁用 evidence 时 records 字段是否仍完整；后者用于测试 calibration / threshold / decision / table builder 是否可运行。

```text
artifacts/run_manifest.json

tables/ablation_table.csv
```

### （四）第四步：实现接口层

阶段 0 只冻结接口，不追求真实方法性能。推荐定义如下接口。

```python
class LatentBackend:
    def build_sample(self, sample_id: str, split: str, sample_role: str) -> LatentSample:
        ...
```

```python
class WatermarkMethod:
    def embed(self, sample: LatentSample, payload: dict) -> EmbeddedSample:
        ...

    def detect(self, sample: LatentSample, threshold: Threshold | None) -> DetectionResult:
        ...

    def detect_batch(self, samples: list[LatentSample], threshold: Threshold | None) -> list[DetectionResult]:
        ...
```

```python
class EvidenceExtractor:
    def extract(self, sample: LatentSample) -> EvidenceScores:
        ...
```

阶段 0 的 `EvidenceExtractor` 允许存在两种实现：

```text
empty_evidence_extractor_placeholder
random_evidence_extractor_random
```

二者必须都输出完整结构：

```json
{
  "S_tubelet": "float|null",
  "S_sync": "float|null",
  "S_traj": "float|null",
  "S_final": "float"
}
```

不得因为 trajectory 未实现而删除 `S_traj` 字段。项目构建文件也明确要求 disabled evidence 必须显式记录为 `null`，不能通过缺失字段表示禁用状态。

### （五）第五步：实现随机 latent backend

阶段 0 的 backend 推荐命名为：

```text

```

```text

probe_backend_p0
```

其输出字段建议为：

```json

  "latent_generation_seed_random": 12345,
```

这里 `latent_tensor_digest_random` 与 `latent_generation_seed_random` 必须带 `_random` 后缀，避免后续被误当成真实视频 latent 证据。

### （六）第六步：实现 threshold calibration

`ThresholdCalibrator` 只能读取：

```text
split == calibration
sample_role in {clean_negative, attacked_negative}
score_name == S_final
```

阈值定义应遵循 fixed low-FPR protocol。方法机制中已明确给出阈值应由 calibration negative 的 (S_{\mathrm{final}}) 上分位数确定，并在 test split 上固定使用；同时，calibration negative 必须同时包含 clean negative 与 attacked negative，以避免低估 synchronization search 在 attacked negative 中产生的偶然相关峰。

推荐 threshold record 如下：

```json
{
  "threshold_id": "fixed_low_fpr_calibrated_detection:S_final:random_score_detector_random",
  "run_id": "string",
  "method_family": "temporal_tubelet_watermark",
  "method_variant": "random_score_detector_random",
  "score_name": "S_final",
  "target_fpr": 0.001,
  "calibration_split": "calibration",
  "calibration_negative_roles": [
    "clean_negative",
    "attacked_negative"
  ],
  "threshold_value": 2.315,
  "threshold_quantile": 0.999,
  "num_calibration_negatives": 1000,
  "threshold_source_record_digest": "sha256...",
  "fusion_rule": "random_score_linear_fusion_random",
  "created_at": "ISO-8601"
}
```

需要特别注意：`threshold_id` 不建议加入 `_v1` 或 `_p0`。它应由语义组成：

```text
<protocol_name>:<score_name>:<method_variant>
```

### （七）第七步：实现 ProtocolRunner

`ProtocolRunner` 的执行顺序必须固定：

```text
1. 读取配置；
2. 创建 run_id 与输出目录；
3. 构建 split；
4. 构建 event plan；
5. 对 calibration negative 运行 detect without threshold；
6. 写入 calibration score records；
7. 使用 calibration negative 生成 threshold；
8. 写入 threshold record；
9. 对 test split 固定 threshold 运行 detect；
10. 写入 test score records；
11. 从 records 生成 metrics 与 tables；
12. 写入 run manifest。
```

其中，第 7 步之后禁止 test 阶段写入新 threshold。项目构建文件明确要求 test 阶段不能写入新的 threshold，且所有 ablation variant 必须共享同一 split 与 attack matrix。

## 六、阶段 0 的测试与审计清单

### （一）`test_records_schema.py`

必须验证：

```text
event_scores.jsonl 中每条记录都包含 split、sample_role、method_variant、attack_name、evidence_scores、S_final、decision；
S_tubelet、S_sync、S_traj、S_final 四个字段必须存在；
disabled evidence 必须以 null 表示，不得缺失字段；
placeholder_fields 与 random_fields 必须存在。
```

### （二）`test_protocol_no_threshold_leakage.py`

必须验证：

```text
threshold_source_record_digest 只能覆盖 calibration negative records；
test split records 不得出现在 threshold_source_record_digest 中；
test positive 不得参与 threshold_value、fusion_rule 或 score normalization 的生成。
```

### （三）`test_ablation_consistency.py`

必须验证：

```text
所有 method_variant 使用同一 split plan；
所有 method_variant 使用同一 attack matrix；
所有 method_variant 使用同一 target_fpr；
不同 variant 不允许各自修改 sample role 定义。
```

### （四）`test_table_rebuild.py`

必须验证：

```text
删除 tables/main_metrics.csv；
重新运行 TableBuilder；
新表格 digest 与原 tables_digest 一致。
```

### （五）`test_placeholder_random_field_audit.py`

必须验证：

```text
所有包含 placeholder 语义的字段名必须以 _placeholder 结尾；
所有由 PRNG 或模拟分数生成的字段名必须以 _random 结尾；
records、thresholds、manifest 中 placeholder_fields 与 random_fields 能覆盖相应字段。
```

### （六）`test_no_stage_weak_naming.py`

必须验证新增字段名不得匹配：

```regex
.*_v[0-9]+$
.*_p[0-9]+$
.*stage[0-9]+.*
.*probe[0-9]+.*
```

允许例外：

```text
兼容历史 schema 的 project_stage 字段；
第三方库固定字段；
已有历史文件中未被本次修改的字段。
```
