# 分阶段构建流程：面向主流正式会议或期刊投稿的显式同步 tubelet 水印论文路线

## 一、文档定位

本文档定义从当前 `synthetic_tubelet_sync_probe` 到投稿级 `full_paper_protocol` 的分阶段构建流程。目标不是仅完成工程 smoke，而是最终产出能够支撑主流正式会议或期刊投稿的完整证据包，包括真实视频 VAE formal、H.264 / H.265 压缩攻击、空间攻击、真实视频重编码链路、LPIPS 等质量指标、外部视频水印 baseline、投稿级图表、claim audit 和复现包。

本文档采用项目现有语义阶段名称，不引入弱阶段编号、版本后缀或数字阶段别名。所有正式 method variant、字段、表格、报告和输出路径必须继续遵守 `.codex/project_contract.md`、`docs/file_organization.md`、`docs/test_case_constraints.md`、`docs/field_registry.md` 与 harness audit 的治理要求。

---

## 二、总体目标

最终论文目标定义为：

```text
证明 temporal_synchronized_tubelet_code 在 fixed_low_fpr_calibrated_detector 下，
能够在真实视频 VAE latent、压缩攻击、空间攻击、时间攻击和 local clip detection 中
保持鲁棒检测能力，并通过 aligned_payload_safety 机制控制 negative FPR 风险。
```

投稿级主张必须同时满足以下条件：

1. 真实视频经 VAE encode / decode / attack / re-encode 后仍能完成检测；
2. `tubelet_sync` 相对 `tubelet_only` 在 temporal crop 或 local clip 中有可复现增益；
3. clean negative 与 attacked negative FPR 均受 fixed-FPR 协议控制；
4. H.264 / H.265、spatial resize、crop resize、blur、Gaussian noise、frame dropping、speed change 和 local clip 均有正式记录；
5. PSNR、SSIM、LPIPS、flicker score、motion consistency 或等价 motion proxy 有正式表格支撑；
6. 外部视频水印 baseline 使用同一 split、attack matrix、target FPR 和 table builder 进行对照；
7. 所有 supported claim 均绑定 records、thresholds、tables、curves、figures、reports 或 manifests；
8. 所有产物可从 governed records、thresholds、manifests 和 configs 重建；
9. 复现包包含 records、thresholds、tables、figures、reports、manifests、summary、checks 和 logs；
10. 默认 pytest、formal gate、package checks 和 `python tools/harness/run_all_audits.py` 均通过。

## 三、正式方法命名与机制边界

### （一）正式 method variant

正式内部方法变体保持为：

```text
frame_prc
tubelet_only
tubelet_sync
```

`aligned_payload_safety` 不作为新的正式 method variant。它是 `tubelet_sync` 的机制协议，应记录到：

```text
mechanism_trace.stage2_mechanism_protocol = aligned_payload_safety
mechanism_trace.sync_confidence_gate_rule = aligned_payload_safety_gate
artifacts/stage2_mechanism_decision.json
tables/aligned_payload_safety_audit.csv
reports/stage2_mechanism_audit_report.md
reports/paper_claim_audit.md
```

### （二）外部 baseline 命名

外部 baseline 不得使用 `default`、`best`、`new`、`old`、`full` 或 `final_method` 等无机制语义名称。推荐使用机制语义名称，例如：

```text
external_frame_repetition_watermark
external_spread_spectrum_video_watermark
external_frequency_domain_video_watermark
external_learned_video_watermark
```

如果外部 baseline 无法合法复现或许可证不允许集成，只能进入 `baseline_limitation_report.md`，不能作为 supported claim 的对照证据。

## 四、字段命名统一规则

### （一）sample role

正式 sample role 使用项目代码中的命名：

```text
clean_negative
attacked_negative
watermarked_positive
attacked_positive
```

文档和新增表格应优先使用：

```text
watermarked_positive_TPR
attacked_positive_TPR
clean_negative_FPR
attacked_negative_FPR
```

若历史 table builder 当前仍输出 `clean_positive_TPR`，该字段只能作为兼容别名，论文正文应解释为：

```text
clean_positive_TPR = 未攻击 watermarked_positive 的检测率
```

后续表格 schema 更新时，应将新增规范字段命名为 `watermarked_positive_TPR`，并在过渡期保留 `clean_positive_TPR` 作为 legacy alias。

### （二）evidence score

显式同步论文中的正式 evidence score 使用：

```text
evidence_scores.S_tubelet
evidence_scores.S_sync
evidence_scores.S_traj
evidence_scores.S_final
```

在 `synthetic_tubelet_sync_probe` 和 `real_video_vae_latent_probe` 中：

```text
S_tubelet != null
S_sync 在 tubelet_sync 中 != null
S_traj == null
disabled_evidence includes trajectory
```

禁止把 VAE reconstruction difference、motion proxy、state-space score 或 sampling-time statistic 写入 `S_traj`。

### （三）aligned payload safety 字段

新增或规范化字段应使用以下命名：

```text
S_payload_unaligned
S_payload_aligned
S_payload_rescue_gain
sync_alignment_coverage_ratio
sync_alignment_matched_count
sync_candidate_score_raw
sync_candidate_score_penalized
sync_estimated_offset
sync_estimated_scale
sync_search_space_size
sync_rescue_applied
negative_rescue_over_threshold_count
negative_rescue_over_threshold_rate
aligned_payload_negative_safety_status
```

所有 placeholder 字段必须以 `_placeholder` 结尾，所有 random trace 字段必须以 `_random` 或 `_digest_random` 结尾。

## 六、阶段一：`synthetic_tubelet_sync_probe`

### （一）目标

该阶段用于证明显式同步机制在 synthetic video latent 与 governed temporal attack matrix 中成立。它是进入真实视频 VAE formal 的前置条件。

### （二）允许内容

```text
synthetic video latent
temporal attack matrix
frame_prc
tubelet_only
tubelet_sync
aligned_payload_safety mechanism trace
fixed-FPR calibration
records / thresholds / tables / curves / reports rebuild
```

### （三）禁止内容

```text
real video VAE formal claim
real DiT backend
Flow Matching backend
real video generation
trajectory evidence
state-space evidence
sampling-time weak constraint
```

### （四）必须修复的工程问题

正式 profile 入口必须可运行。当前应修复并验证：

```text
tools/harness/run_synthetic_tubelet_sync_profile.py
```

该入口必须调用实际存在的 synthetic ablation runner，不能导入不存在的模块。

### （五）通过 gate

进入下一阶段前必须满足：

```text
primary_stage1_completion_pass = true
primary_strict_target_fpr_pass = true
tubelet_sync_beats_tubelet_only_under_temporal_crop_or_local_clip = true
records_to_tables = true
records_to_curves = true
records_to_report = true
pytest -q pass
python tools/harness/run_all_audits.py pass
```

若 fresh proof 或 formal run 中 `tubelet_sync` 没有超过 `tubelet_only`，则不得推进到真实视频 formal claim，只能继续调试 sync search、coverage penalty、payload rescue gate、attack matrix 或 embedding margin。

## 八、阶段三：`baseline_comparison_gate`

### （一）目标

该 gate 用于构建外部视频水印 baseline 对照，使论文不只是内部 ablation，而是能回答“相对已有方法是否有优势”的审稿问题。

### （二）baseline 选择规则

每个 baseline 必须满足：

1. 许可证允许复现实验或结果复核；
2. 能在同一真实视频数据、同一 split、同一 attack matrix 下运行；
3. 能输出与项目 records schema 可对齐的 detection score 或 decision；
4. 能使用 calibration negative 固定 threshold；
5. 不能使用 test split 调参；
6. 若 baseline 只能运行部分攻击，必须在 limitation report 中记录。

### （三）本项目选定的外部 baseline

本项目投稿级比较固定采用以下三个外部视频水印 baseline：

| 正式 baseline 名称 | 来源类型 | 主要比较目的 |
|---|---|---|
| `external_videoseal` | 强神经视频水印 baseline | 对比当前较强的视频水印方法在压缩、空间攻击、时间攻击和 local clip 下的鲁棒性 |
| `external_rivagan` | 经典神经视频水印 baseline | 对比经典深度视频水印方法在真实视频攻击链路中的检测率、质量和运行开销 |
| `external_hidden_framewise` | 图像水印逐帧迁移 baseline | 回答“为什么不直接逐帧使用图像水印”的审稿问题，突出显式时间同步的独立价值 |

这三个 baseline 均不得作为项目内部主方法 `method_variant` 替代 `frame_prc`、`tubelet_only` 或 `tubelet_sync`。它们应进入 baseline comparison 子协议，并通过统一 adapter 写入 baseline records。若某个 baseline 因许可证、模型权重、运行环境或输出接口限制无法正式复现，必须在 `reports/baseline_limitation_report.md` 中记录，并不得用该 baseline 支撑“优于该方法”的 supported claim。

### （四）baseline 构建步骤

每个外部 baseline 必须按以下步骤接入：

```text
1. 完成 license / model / code availability 复核。
2. 固定 baseline source digest、model digest、runtime environment 和 dependency manifest。
3. 实现统一 adapter：embed(video, payload, key) -> watermarked_video。
4. 实现统一 adapter：detect(video, key) -> score / confidence / bit_accuracy / decision。
5. 将 baseline 输出映射到项目 records schema，至少保留 baseline_score、decision、failure_reason、quality_metrics、temporal_metrics 和 runtime_metrics。
6. 使用同一 dev / calibration / test split，不允许 baseline 使用自己的 test threshold。
7. 使用 calibration negative 为每个 baseline 和 target FPR 校准 threshold。
8. 在 test split 上运行同一 attack matrix。
9. 由正式 table builder 生成 baseline_comparison_table.csv、baseline_attack_breakdown.csv 和 baseline_runtime_table.csv。
10. 将 baseline 限制、失败样例和不可覆盖攻击写入 baseline_limitation_report.md。
```

三个 baseline 的具体适配要求如下：

#### 1. `external_videoseal`

`external_videoseal` 作为强视频水印 baseline，必须优先覆盖：

```text
H.264 compression
H.265 compression
spatial_resize
crop_resize
temporal_crop
frame_dropping
speed_change
local_clip
```

若其 detector 输出 bit accuracy、confidence 或 binary decision，而不是项目所需连续 `baseline_score`，adapter 必须定义可审计的 score 映射规则，并将原始 detector 输出同时写入 `mechanism_trace` 或 baseline trace 字段。

#### 2. `external_rivagan`

`external_rivagan` 作为经典神经视频水印 baseline，必须重点报告：

```text
attacked_positive_TPR
attacked_negative_FPR
watermarked_positive_TPR
quality_metrics
runtime_metrics
```

若原始实现仅支持有限分辨率、有限帧数或有限 codec，必须在 baseline limitation 中显式记录，不得手工删除失败样本来抬高结果。

#### 3. `external_hidden_framewise`

`external_hidden_framewise` 作为图像水印逐帧迁移 baseline，必须明确其适配方式：

```text
逐帧 embed
逐帧 detect
跨帧 score aggregation
video-level threshold calibration
```

该 baseline 的主要作用是证明没有显式 temporal synchronization 的逐帧图像水印迁移在 temporal crop、frame dropping、speed change 和 local clip 下存在结构性劣势。

### （五）baseline records 字段

baseline records 至少需要包含：

```text

event_id
sample_id
split
sample_role
baseline_name
baseline_family
attack_name
attack_params
target_fpr
threshold_id
baseline_score
baseline_raw_detector_output

failure_reason
quality_metrics
temporal_metrics
runtime_metrics
baseline_trace
```

其中，`baseline_trace` 必须记录 source digest、model digest、adapter version、score mapping rule 和 unsupported attack reason。若某字段只是占位，必须以 `_placeholder` 结尾，且不得进入 supported claim。

### （六）必须产出

```text
tables/baseline_comparison_table.csv
tables/baseline_attack_breakdown.csv
tables/baseline_runtime_table.csv
tables/baseline_threshold_table.csv
reports/baseline_protocol_report.md
reports/baseline_limitation_report.md
figures/baseline_comparison.pdf
```

### （七）baseline gate

通过条件为：

```text

external_rivagan_reported = true
external_hidden_framewise_reported = true
baseline_threshold_protocol_pass = true
baseline_attack_matrix_coverage_reported = true
baseline_runtime_reported = true
baseline_limitations_disclosed = true
baseline_records_to_tables = true
baseline_records_to_report = true
```

如果只能完成一个外部 baseline，则可以投稿 workshop 或技术报告，但不建议作为主流正式会议或期刊的完整投稿版本。如果只能完成两个 baseline，必须在投稿前由 `paper_claim_audit.md` 降级所有“相对外部方法全面优越”的表述。

## 十二、推荐执行顺序

推荐按以下顺序执行：

```text
1. 修复 synthetic profile 入口导入错误。
2. 重新运行 synthetic proof / formal，直到显式同步核心 gate 通过。
3. 将 aligned_payload_safety 固化为 tubelet_sync 的机制协议字段。
4. 统一 records、tables 和 reports 中的 positive 字段命名。
5. 更新 real_video_vae_latent_probe 的 formal gate，使其允许真实视频 VAE、H.264 / H.265、LPIPS 和真实视频质量表。
6. 运行真实视频 VAE formal，并冻结 records、thresholds、tables、figures、reports 和 manifests。
7. 依次接入 `external_videoseal`、`external_rivagan` 和 `external_hidden_framewise`，并为每个 baseline 完成 license / model / source digest 复核。
```
