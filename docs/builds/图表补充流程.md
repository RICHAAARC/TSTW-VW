# 图表补充流程

本文档用于指导 `paper_artifact_gate` 之后、`submission_readiness_gate` 之前的论文图表补充工作。目标是补齐主流正式会议或期刊投稿中常见的鲁棒性曲线、时间质量指标和附加数据集验证图表，并保持所有图表可由受治理 records、tables、manifests 重建。

## 1. 总体目标

当前 `paper_artifact_gate` 已经能够产出主方法比较、攻击分解、同步增益、外部 baseline 比较、ROC / AUC、PSNR / SSIM / LPIPS、正式计分效率、视觉样例和 claim audit。后续仍需要补充以下投稿增强图表:

1. 攻击强度曲线: H.264、H.265、`temporal_crop`、`frame_dropping`。
2. 时间质量指标: `t-LPIPS`、`t-SSIM`。
3. 附加数据集验证: UCF101 subset, 仅运行内部 3 个正式方法 `frame_prc`、`tubelet_only`、`tubelet_sync`。

这三类补充实验不应覆盖已经冻结的 `real_video_vae_latent_probe` 和 `baseline_comparison_gate` 结果。正确做法是新增受治理 workflow, 再由 `paper_artifact_gate` 汇总为论文图表。

## 2. 新增 workflow

建议新增 3 个补充 workflow:

```text
attack_strength_curve_probe
temporal_quality_metric_probe
additional_dataset_validation_probe
```

对应 Google Drive 结果目录:

```text
TSTW/results/attack_strength_curve_probe/
TSTW/results/temporal_quality_metric_probe/
TSTW/results/additional_dataset_validation_probe/
```

每个 workflow 仍然遵守 shard run 和 shard aggregation 拆分原则:

```text
TSTW/results/<WORKFLOW_KEY>/shard_runs/
TSTW/results/<WORKFLOW_KEY>/shard_aggregated/
```

对于 UCF101 附加数据集, 建议增加数据集名层级:

```text
TSTW/results/additional_dataset_validation_probe/ucf101/shard_runs/
TSTW/results/additional_dataset_validation_probe/ucf101/shard_aggregated/
```

## 3. 推进顺序

推荐顺序如下:

1. `temporal_quality_metric_probe`
2. `attack_strength_curve_probe`
3. `additional_dataset_validation_probe`
4. 更新 `paper_artifact_gate` 汇总逻辑
5. 更新 claim audit 和 submission gap audit

主要原因是 `temporal_quality_metric_probe` 可以优先复用已经落盘的阶段二视频结果, 成本最低; `attack_strength_curve_probe` 需要补跑多强度攻击; `additional_dataset_validation_probe` 需要新增数据集处理和正式验证, 成本最高。

## 4. temporal_quality_metric_probe

### 4.1 目标

补充视频时间一致性质量指标:

```text
t-LPIPS ↓
t-SSIM ↑
```

该流程回答的问题是: 水印嵌入、VAE decode / re-encode 和攻击链路是否破坏视频相邻帧的一致性。

### 4.2 输入

优先读取已经完成的阶段二 shard run:

```text
TSTW/results/real_video_vae_latent_probe/shard_runs/<RUN_ID>/compat_run_root/
```

需要读取:

```text
records/event_scores.jsonl
artifacts/videos/source/
artifacts/videos/decoded/
artifacts/videos/attacked/
```

### 4.3 方法与样本范围

方法范围:

```text
frame_prc
tubelet_only
tubelet_sync
```

视频角色建议包含:

```text
source
decoded_watermarked
attacked
```

攻击范围建议至少包含:

```text
no_attack
h264_compression
h265_compression
temporal_crop
frame_dropping
```

### 4.4 指标定义

`t-LPIPS` 建议定义为相邻帧 LPIPS 的平均值:

```text
t-LPIPS(video) = mean_t LPIPS(frame_t, frame_{t+1})
```

`t-SSIM` 建议定义为相邻帧 SSIM 的平均值:

```text
t-SSIM(video) = mean_t SSIM(frame_t, frame_{t+1})
```

如果使用 LPIPS 模型, 需要记录模型名称、权重来源、模型 digest 和运行设备。若 LPIPS 在本地不可用, 可以先实现 `t-SSIM` 和结构化缺失说明, 但不能把缺失的 `t-LPIPS` 标记为 supported。

### 4.5 输出文件

shard run 输出:

```text
records/temporal_quality_records.jsonl
artifacts/temporal_quality_runtime_manifest.json
```

shard aggregation 输出:

```text
tables/temporal_quality_metric_table.csv
figure_data/temporal_quality_metric_figure_data.csv
artifacts/temporal_quality_metric_manifest.json
claim_audit/temporal_quality_claim_audit.csv
```

建议字段:

```text
method_name
attack_name
sample_id
video_source_id
video_role
frame_count
mean_t_lpips
std_t_lpips
mean_t_ssim
std_t_ssim
lpips_model_name
lpips_model_digest
source_video_path
decoded_video_path
attacked_video_path
source_artifact
```

### 4.6 完成标准

该流程完成需要满足:

1. 输出 records、tables、figure_data、manifest。
2. 至少覆盖 3 个内部正式方法。
3. 至少覆盖 H.264、H.265、`temporal_crop`、`frame_dropping`。
4. 指标可以从 mp4 重新计算。
5. `paper_artifact_gate` 可以读取聚合结果。
6. claim audit 明确 `t-LPIPS` 和 `t-SSIM` 只支持时间质量 claim, 不支持检测鲁棒性 claim。

## 5. attack_strength_curve_probe

### 5.1 目标

补充攻击强度曲线:

```text
TPR@1%FPR vs attack strength
AUC vs attack strength
```

该流程回答的问题是: 攻击强度逐步增加时, `tubelet_sync` 是否仍然比 `tubelet_only` 和 `frame_prc` 更稳健。

### 5.2 攻击强度配置

建议第一版只覆盖 4 类关键攻击:

#### H.264

```text
crf = 18, 23, 28, 33, 38
```

#### H.265

```text
crf = 18, 23, 28, 33, 38
```

#### temporal_crop

```text
keep_ratio = 0.90, 0.75, 0.60, 0.50
```

#### frame_dropping

```text
drop_rate = 0.10, 0.20, 0.30, 0.40, 0.50
```

总攻击强度点数为 19 个。

### 5.3 方法范围

第一版只运行内部正式方法:

```text
frame_prc
tubelet_only
tubelet_sync
```

不建议第一版加入外部 baseline, 因为外部 baseline 的高强度 sweep 成本较高, 且会显著增加 Colab 分片运行时间。

### 5.4 fixed-FPR 规则

必须保持当前固定低 FPR 协议:

1. 阈值只能由 calibration split 的 negatives 校准。
2. test split 不能参与阈值选择。
3. 不允许为每个 test attack strength 单独调阈值。
4. 所有强度点应复用同一 method-level fixed-FPR 校准规则, 或明确记录 calibration strength policy。

推荐策略:

```text
calibration negatives 使用所有目标攻击强度的 calibration negatives 合并校准。
test split 按 attack_name 和 attack_strength 分组报告 TPR / FPR / AUC。
```

### 5.5 输出文件

shard run 输出:

```text
records/attack_strength_event_scores.jsonl
thresholds/attack_strength_thresholds.json
artifacts/attack_strength_runtime_manifest.json
```

过渡实现允许先使用 `from_stage_two_existing_records` 模式从已经冻结的阶段二真实视频 VAE records 生成基础曲线 records:

```text
TSTW/results/attack_strength_curve_probe/shard_runs/attack_strength_curve_probe_base_records_<UTC_TIME>_<SHORT_COMMIT>/
  records/attack_strength_event_scores.jsonl
  artifacts/attack_strength_base_records_manifest.json
```

该模式的作用是先打通“基础 records -> 聚合表 -> 阶段四论文图表”的链路。由于它没有重新运行完整多强度攻击 sweep, 其 shard manifest 必须将 `claim_support_allowed` 标记为 `false`, 后续投稿 claim 仍应由真实多强度 runner 的 records 支撑。

shard aggregation 输出:

```text
tables/attack_strength_tpr_table.csv
tables/attack_strength_auc_table.csv
figure_data/attack_strength_curve_figure_data.csv
artifacts/attack_strength_curve_manifest.json
claim_audit/attack_strength_claim_audit.csv
```

建议字段:

```text
method_name
attack_name
attack_strength_name
attack_strength_value
target_fpr
threshold
positive_count
negative_count
tpr_at_target_fpr
fpr_at_threshold
auc
source_artifact
```

### 5.6 完成标准

该流程完成需要满足:

1. H.264、H.265、`temporal_crop`、`frame_dropping` 均完成强度 sweep。
2. 每个强度点均包含 `frame_prc`、`tubelet_only`、`tubelet_sync`。
3. 输出 TPR 曲线和 AUC 曲线所需的 figure_data。
4. 无 threshold leakage。
5. claim audit 能确认 `tubelet_sync` 在时间攻击强度升高时相对 `tubelet_only` 的优势是否成立。

## 6. additional_dataset_validation_probe

### 6.1 目标

补充 UCF101 subset 泛化验证, 用于支持方法不只依赖 DAVIS 2017 数据分布。

目标 claim:

```text
显式同步机制在额外真实视频数据集上仍保持相对优势。
```

### 6.2 数据集范围

推荐使用 UCF101 subset, 第一版不跑全量。

建议规模:

```text
100 到 300 个 clips
32 frames
256x256
8 fps
```

split 建议:

```text
dev: 1/3
calibration: 1/3
test: 1/3
```

### 6.3 数据集输出目录

建议新增 processed dataset:

```text
TSTW/datasets/processed/real_video_vae_latent_probe_ucf101_256x256_32f_8fps_paper_low_fpr/
```

该目录应包含:

```text
dataset_manifest.json
processed_dataset_summary.json
checks/processed_dataset_checks.json
source/*.mp4
```

### 6.4 方法与攻击范围

方法范围:

```text
frame_prc
tubelet_only
tubelet_sync
```

攻击矩阵第一版建议使用主实验固定攻击矩阵的子集:

```text
no_attack
h264_compression
h265_compression
crop_resize
temporal_crop
frame_dropping
local_clip
```

如果算力允许, 再扩展到完整 DAVIS 主实验攻击矩阵。

### 6.5 输出文件

shard run 输出:

```text
records/additional_dataset_event_scores.jsonl
thresholds/additional_dataset_thresholds.json
artifacts/additional_dataset_runtime_manifest.json
```

shard aggregation 输出:

```text
tables/additional_dataset_main_tpr_fpr_table.csv
tables/additional_dataset_attack_breakdown_table.csv
tables/additional_dataset_quality_table.csv
figure_data/additional_dataset_comparison_figure_data.csv
artifacts/additional_dataset_validation_manifest.json
claim_audit/additional_dataset_claim_audit.csv
```

建议字段:

```text
dataset_name
dataset_subset_id
method_name
attack_name
target_fpr
positive_count
negative_count
tpr_at_target_fpr
fpr_at_threshold
auc
source_artifact
```

### 6.6 完成标准

该流程完成需要满足:

1. UCF101 subset processed dataset 构建完成并通过 checks。
2. 3 个内部正式方法全部完成。
3. fixed-FPR 校准协议与主实验一致。
4. 至少输出 main TPR/FPR 表和 attack breakdown 表。
5. 结果可以由 `paper_artifact_gate` 汇总。
6. 若 `tubelet_sync > tubelet_only > frame_prc` 不成立, 必须在 claim audit 中记录不可支持或可解释例外。

## 9. 本地与 Colab 分工

### 9.1 本地可完成

本地可以先完成:

1. workflow contract 和配置文件。
2. runner / aggregator 脚本骨架。
3. notebook 骨架。
4. 约束测试。
5. `paper_artifact_gate` 读取接口。
6. 小型 fixture 的表格和 claim audit 测试。

### 9.2 需要 Colab 的部分

以下部分建议在 Colab 执行:

1. 使用 GPU 计算 `t-LPIPS`。
2. 多强度攻击下的 VAE encode / decode / re-encode。
3. UCF101 subset 正式 run。
4. 大规模 shard run 和 shard aggregation。

如果只计算 `t-SSIM`, 可以尝试本地运行; 但为保持流程一致, 仍建议使用 Colab 跑正式结果包。

## 10. 最小可行推进版本

如果需要先快速形成投稿增强版, 可以采用以下最小配置:

### 10.1 攻击强度曲线

```text
methods = tubelet_only, tubelet_sync
h264 crf = 23, 28, 33, 38
h265 crf = 23, 28, 33, 38
temporal_crop keep_ratio = 0.90, 0.75, 0.60
frame_dropping drop_rate = 0.10, 0.30, 0.50
```

正式投稿版再补回 `frame_prc`。

### 10.2 时间质量

```text
source vs decoded_watermarked
metrics = t-LPIPS, t-SSIM
```

### 10.3 UCF101 subset

```text
clip_count = 100
methods = frame_prc, tubelet_only, tubelet_sync
attacks = no_attack, h264_compression, h265_compression, temporal_crop, frame_dropping
```

## 11. 风险与约束

1. 不得把缺失的 FID / FVD / BLIP 标记为 supported。
2. 不得从固定攻击协议结果外推攻击强度曲线。
3. 不得使用 test split 调阈值。
4. 不得把 notebook 作为唯一正式逻辑来源。
5. 不得覆盖已冻结的阶段二、阶段三结果包。
6. 所有新增字段若进入正式 records 或 claim audit, 需要同步更新字段治理文档。
7. 所有正式图表必须由 records、tables、figure_data 和 manifest 重建。

## 12. 推荐下一步

建议先推进 `temporal_quality_metric_probe`, 因为它复用现有阶段二 shard run 视频, 工程风险最低。完成后再推进 `attack_strength_curve_probe`, 最后推进 UCF101 subset 附加数据集验证。
