## 1. 阶段定位

`trajectory_statistic_probe` 是项目的第三个语义构建阶段。该阶段的目标不是接入完整的 DiT / Flow Matching 视频生成模型, 也不是直接实现 sampling-time watermark embedding, 而是在已经完成的 `real_video_vae_latent_probe` 阶段 2 机制证明基础上, 引入第三类证据 `trajectory_evidence`, 验证 detector 侧的 trajectory statistic 是否能在 fixed low-FPR protocol 下提供独立、可审计、可复现的边际增益。

本阶段要回答的核心问题是:

```text
在 calibration-only 阈值协议和 test-only final audit 约束下,
S_traj 是否能作为独立 evidence 写入 records,
并使 traj_only / tubelet_traj / tubelet_sync_trajectory_fusion 在不破坏 negative FPR 的前提下,
相对已冻结的 tubelet_only 或 tubelet_sync baseline 产生可解释的边际增益。
```

本阶段只证明以下内容:

```text
trajectory observation 可以由受治理输入构造;
S_traj 可以作为独立 evidence 写入统一 event_score records;
trajectory-enabled method variants 可以与已有 protocol runner / calibrator / table builder 协同运行;
trajectory-disabled baseline 必须保持 S_traj == null;
trajectory evidence 的 gain、correlation、control、leakage 和 runtime overhead 均可审计;
所有 tables / reports / mechanism decision 均可由 records 与 thresholds 重建。
```

本阶段不证明以下内容:

```text
真实 DiT / Flow Matching 生成过程已接入;
真实 Flow Matching sampling trajectory 已完全可观测;
trajectory-aware sampling-time embedding 已经成立;
trajectory evidence 必然进入论文主方法;
阶段 4 trajectory_aware_sampling_probe 可以自动启动。
```

## 4. 方法机制边界

阶段 3 必须保持以下机制边界。

### 4.1 允许的 evidence 关系

```text
S_tubelet: 来自 tubelet evidence;
S_sync: 来自阶段 2 已固化的 sync evidence;
S_traj: 来自 trajectory observation 和 trajectory statistic;
S_final: 由受治理 fusion rule 组合已有 evidence 得到。
```

`S_traj` 必须满足:

```text
不能由 sample_role 推导;
不能由 positive-only artifact 推导;
不能读取 watermarked_latent_relpath 作为 detector 侧输入;
不能通过 test split 调参;
不能写回阶段 2 records;
不能替代阶段 2 的 aligned_payload_safety gate。
```

### 4.2 method variants

阶段 3 的正式 method variants 为:

```text
tubelet_only
tubelet_sync
traj_only
tubelet_traj
tubelet_sync_trajectory_fusion
```

含义如下:

1. `tubelet_only`: 阶段 1 / 阶段 2 已有 tubelet baseline, `S_traj == null`。
2. `tubelet_sync`: 阶段 2 frozen sync baseline, `S_traj == null`。
3. `traj_only`: 只启用 trajectory evidence, 用于判断 `S_traj` 是否有独立判别力。
4. `tubelet_traj`: 启用 tubelet + trajectory, 不启用 sync, 用于判断 trajectory 相对 tubelet 的边际增益。
5. `tubelet_sync_trajectory_fusion`: 启用 tubelet + sync + trajectory, 用于判断 trajectory 相对 frozen tubelet_sync 的边际增益。

禁止新增只表达优劣、新旧、默认状态或完整程度而不表达机制组成的弱语义 method variant。

### 4.3 trajectory source 分级

阶段 3 支持的 trajectory source 应分级解释:

```text
synthetic_flow_trajectory: 仅用于 unit / smoke / sanity check;
latent_interpolation_surrogate: 可用于 scaffold 工程闭环, 不可单独支撑 mechanism PASS;
vae_reencoded_endpoint_surrogate: 可用于阶段 2 frozen replay 的低成本过渡验证;
fm_inversion_candidate: 可作为阶段 3 formal mechanism audit 的候选正式来源;
fm_backend_candidate: 保留给后续更高阶段, 不在当前阶段直接接入真实生成后端。
```

如果只使用 surrogate source, 则 `Stage3ImplementationDecision` 可以为 `PASS`, 但 `Stage3MechanismDecision` 不应为 `PASS`。

## 7. 决策口径

### 7.1 Stage3ImplementationDecision

`Stage3ImplementationDecision == PASS` 表示阶段 3 工程链路闭合。它至少要求:

```text
records 非空;
thresholds 非空;
trajectory-enabled variants 均有非空 S_traj;
trajectory-disabled variants 均保持 S_traj == null;
trajectory_source_kind / trajectory_statistic_kind / trajectory_time_grid 完整;
tables / reports 可由 records 重建;
runtime overhead 可报告;
manifest 和 runtime_config 可追溯;
阶段 2 records 未被修改。
```

### 7.2 Stage3MechanismDecision

`Stage3MechanismDecision == PASS` 表示 trajectory evidence 形成机制证明。它至少要求:

```text
Stage2DependencyStatus == PASSED;
trajectory source 满足 formal source 要求;
trajectory-enabled method 相对 frozen baseline 的 delta_traj > 0;
clean_negative_FPR 和 attacked_negative_FPR 不超过 gate;
S_traj 与 S_tubelet / S_sync 不是高度冗余;
trajectory controls 不能复现主增益;
增益可以定位到 attack 或 failure subset;
runtime overhead 已报告且不阻断 formal protocol;
所有结论均可由 records / thresholds / tables / reports 支撑。
```

如果只满足 implementation, 但 source、gain、correlation、control 或 leakage 不满足要求, 则:

```text
Stage3ImplementationDecision == PASS
Stage3MechanismDecision == FAIL 或 INCONCLUSIVE
NextAllowedStageByTrajectory != trajectory_aware_sampling_probe
```

## 8. 失败与阻塞处理

常见 blocking reasons 与处理方式如下:

```text
real_video_vae_latent_dependency_not_passed:
  先修复 frozen baseline loader 或确认阶段 2 final formal package。

formal_trajectory_source_missing:
  先补充 fm_inversion_candidate 或正式 replay source, 不要用 surrogate 强行 PASS。

s_traj_missing_for_trajectory_enabled_variant:
  检查 method config、trajectory backend 和 method_factory。

s_traj_non_null_for_trajectory_disabled_variant:
  检查 baseline method config, 防止阶段 2 baseline 被 trajectory 污染。

trajectory_gain_not_positive:
  检查 statistic kind、trajectory_weight 和 replay source。

trajectory_score_highly_correlated_with_tubelet:
  检查 S_traj 是否退化为 tubelet projection 的重复表达。

trajectory_control_matches_main_gain:
  检查 key-conditioned trajectory statistic 是否失效。

trajectory_negative_leakage_exceeds_gate:
  优先修复 calibration 和 fusion, 不允许使用 test split 调参。
```

### 20.2 根因分析

本次失败不是阶段 2 frozen package 问题。阶段 2 输入已经通过 frozen baseline loader:

```text
Stage2FrozenBaselineManifest == PASSED
Stage2ImplementationDecision == PASS
```

实际根因分为四类:

1. `trajectory_gain_gate` 只检查 attacked-positive TPR 增量。formal replay 中 `tubelet_only` 与 `tubelet_sync` baseline 的 attacked-positive TPR 已经达到 1.0, 二值 TPR 无法继续上升, 因此 `max_delta_traj == 0.0`。但 residual boost 仍提高了部分正样本相对 threshold 的分数裕量。
2. `trajectory_negative_leakage_gate` 把 `traj_only` ablation 也纳入候选机制 leakage。实际 fused candidates `tubelet_traj` 与 `tubelet_sync_trajectory_fusion` 没有增加负样本泄漏, 误报主要来自 `traj_only + local_clip + attacked_negative`。
3. `trajectory_control_gate` 依赖 `max_delta_traj > 0.0`, 因此在 TPR 饱和时被 gain gate 连带拖失败。control suppression ratio 本身低于阈值。

治理边界如下:

- 本地 runner 仍保持 `runtime_backend_connection_allowed_in_local_runner = false`。
- 外部 smoke 必须记录 `external_real_backend_connection_attempted = true` 和 `external_real_backend_connection_succeeded = true`。
- 外部 smoke 不得记录真实生成越界, 即 `external_real_generation_attempted = false`。
- 外部 smoke 不得记录真实 watermark 集成越界, 即 `external_real_watermark_integration_attempted = false`。

因此当前项目已经具备“接收并判定真实 GPU backend connection smoke 结果”的仓库侧 gate。下一步需要在 Colab 或同等真实 GPU 环境中实际产出上述外部结果摘要, 再由该 gate 判断是否允许进入 `real_backend_runtime_validation_gate`。
