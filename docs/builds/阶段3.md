# 阶段 3：Flow Matching Trajectory Statistic Probe 构建说明

## 一、阶段 3 的定位

阶段 3 应定义为 **Flow Matching Trajectory Statistic Probe**，即“Flow Matching 轨迹统计机制验证”。其核心目标不是接入完整 DiT / Flow Matching 视频生成模型，也不是开始构建 trajectory-aware sampling-time embedding，而是在阶段 1 已验证 `temporal-synchronized tubelet code`、阶段 2 已建立 `real_video_vae_latent_probe` 工程闭环的基础上，引入第三条 evidence：`trajectory_evidence`。

阶段 3 只回答一个核心机制问题：

```text
在 fixed low-FPR protocol 下，Flow Matching trajectory statistic 是否能够提供独立于 tubelet evidence 与 sync evidence 的非冗余边际增益，同时不造成 clean negative FPR 或 attacked negative FPR 失控。
```

因此，阶段 3 的最小结论应为：

```text
本阶段只证明：
trajectory observation 可以从受控 latent trajectory、近似 inversion trajectory 或 surrogate trajectory 中稳定构建；
S_traj 可以作为独立 evidence 写入统一 records；
traj_only、tubelet_traj 与 tubelet_sync_trajectory_fusion 可以在同一 fixed low-FPR protocol 下运行；
S_traj 与 S_tubelet、S_sync 之间不是高度冗余；
trajectory evidence 在至少一种攻击类型或一种失败子集上提供可解释的边际增益；
trajectory evidence 的 clean negative FPR 与 attacked negative FPR 可被 calibration 与 audit 控制；
trajectory reconstruction 与 trajectory scoring 的 runtime overhead 可以被量化并进入报告。
```

阶段 3 不应证明：

```text
DiT / Flow Matching sampling-time watermark embedding 已经有效；
trajectory-aware sampling constraint 已经是主论文必需模块；
完整论文主表、baseline comparison 与投稿级正式协议已经冻结；
阶段 2 的 sync 机制 negative leakage 问题已经自然解决；
真实视频生成模型端到端水印嵌入已经完成。
```

阶段 3 的正确工程含义是：在不破坏阶段 2 `real_video_vae_latent_probe` 的前提下，新增一个可独立运行、可独立审计、可配置关闭的 `trajectory_statistic_probe`，用于验证 detector 侧 trajectory statistic 的统计判别力。

## 六、阶段 3 的算法原语

### （一）trajectory observation

阶段 3 的输入不应只是最终 latent \(z\)，而应构造一组离散 trajectory observations：

\[
\mathcal{T}(x)=\{\hat{z}_{t_1},\hat{z}_{t_2},\ldots,\hat{z}_{t_M}\},
\]

其中，\(x\) 表示待检测视频或其 VAE latent 表示，\(t_m\in[0,1]\) 表示离散 trajectory time grid。

在阶段 2 未完成时，允许使用 surrogate trajectory：

\[
\hat{z}_{t_m}=(1-t_m)\hat{z}_{\mathrm{noise}}+t_m\hat{z}_{\mathrm{obs}},
\]

但该分支只能用于 scaffold，不得作为 formal trajectory 机制结论。

阶段 2 完成后，正式 trajectory 分支应优先使用 inversion candidate：

\[
\hat{z}_{t_m}=I_{\phi}(x,t_m),
\]

其中，\(I_{\phi}\) 表示 detector 侧可复现的 inversion 或 reconstruction procedure。

### （二）velocity projection statistic

阶段 3 主统计量建议定义为 velocity projection：

\[
\hat{v}_{m,g}=\frac{\hat{z}_{t_{m+1},g}-\hat{z}_{t_m,g}}{t_{m+1}-t_m},
\]

\[
S_{\mathrm{traj}}=\frac{1}{|\mathcal{G}|(M-1)}
\sum_{g\in\mathcal{G}}\sum_{m=1}^{M-1}
c_g\langle \hat{v}_{m,g},u_g\rangle。
\]

其中，\(g\) 表示 tubelet group，\(\mathcal{G}\) 表示所有有效 tubelet group，\(c_g\) 表示 key-conditioned tubelet code，\(u_g\) 表示 key-conditioned projection direction。

### （三）displacement projection statistic

作为辅助统计量，可以记录 displacement projection：

\[
S_{\mathrm{disp}}=\frac{1}{|\mathcal{G}|}
\sum_{g\in\mathcal{G}}c_g\langle \hat{z}_{t_M,g}-\hat{z}_{t_1,g},u_g\rangle。
\]

该统计量不得默认替代主 `S_traj`，只能作为 mechanism trace 或 ablation score。

### （四）curvature residual

为了排除 trajectory score 只是终点 latent 差异的重复表达，可记录 curvature residual：

\[
R_{\mathrm{curv}}=\frac{1}{|\mathcal{G}|(M-2)}
\sum_{g\in\mathcal{G}}\sum_{m=2}^{M-1}
\left\|\hat{z}_{t_{m+1},g}-2\hat{z}_{t_m,g}+\hat{z}_{t_{m-1},g}\right\|_2。
\]

`R_curv` 不直接参与 detection threshold，但应进入 `mechanism_trace`，用于解释 trajectory reconstruction 是否退化为线性插值。

## 七、阶段 3 的 method variant 设计

### （一）基础变体

阶段 3 必须支持以下变体：

```text
tubelet_only
tubelet_sync
traj_only
tubelet_traj
tubelet_sync_trajectory_fusion
```

### （二）变体定义

`traj_only`：

```text
enable_tubelet: false
enable_sync: false
enable_trajectory: true
S_final = S_traj
```

`tubelet_traj`：

```text
enable_tubelet: true
enable_sync: false
enable_trajectory: true
S_final = alpha * S_tubelet + (1 - alpha) * S_traj
```

`tubelet_sync_trajectory_fusion`：

```text
enable_tubelet: true
enable_sync: true
enable_trajectory: true
S_final = beta * S_tubelet_sync + (1 - beta) * S_traj
```

其中，\(\alpha\) 与 \(\beta\) 必须在 dev 或 calibration 中确定，不得根据 test positive 结果反向选择。

### （三）不建议在阶段 3 初期实现的变体

阶段 3 初期不建议实现：

```text
trajectory_rescue_gate
sync_then_trajectory_rescue
trajectory_then_sync_rescue
attack_specific_trajectory_fusion
learned_fusion_classifier
```

原因是阶段 2 的 sync rescue leakage 尚未完全收敛。若阶段 3 同时引入复杂 rescue gate，会导致 leakage 来源难以解释。

## 十、trajectory negative control

阶段 3 必须新增 trajectory negative control，用于证明 `S_traj` 不是普通 latent 差异、tubelet score 重复表达或 key 泄漏。

### （一）必须支持的 control

```text

traj_time_reversed
traj_random_projection_control
traj_group_permutation_control
```

### （二）control 判定逻辑

若正式 trajectory 分支有效，应满足：

\[
\Delta_{\mathrm{traj}} > \Delta_{\mathrm{control}}。
\]

其中：

\[
\Delta_{\mathrm{traj}}=\mathrm{TPR}(\mathrm{tubelet\_sync\_trajectory\_fusion})-\mathrm{TPR}(\mathrm{tubelet\_sync})。
\]

若 `traj_shuffled_key`、`traj_time_reversed` 或 `traj_random_projection_control` 也能产生相近增益，则阶段 3 不应判定为 mechanism pass。

## 十二、阶段 3 audit 与 checker

### （一）implementation audit

`Stage3ImplementationDecision` 的通过条件如下：

1. `traj_only`、`tubelet_traj`、`tubelet_sync_trajectory_fusion` 均可运行并写出 records；
2. 启用 trajectory 的 records 中 `S_traj` 非空；
3. 未启用 trajectory 的 records 中 `S_traj == null`；

7. 阶段 1 与阶段 2 的 smoke tests 未被破坏；
8. trajectory 模块可通过配置完全禁用。

### （二）mechanism audit

`Stage3MechanismDecision` 的通过条件如下：

1. `S_traj` 在 positive 与 negative 之间存在统计分离；
2. `tubelet_traj` 或 `tubelet_sync_trajectory_fusion` 相比无 trajectory 版本存在正向边际增益；
3. `S_traj` 与 `S_tubelet`、`S_sync` 不是高度冗余；
4. clean negative FPR 与 attacked negative FPR 不超过阶段 3 gate；
5. trajectory gain 能映射到明确攻击类型或失败模式；
6. trajectory negative control 不产生同等增益；
7. trajectory runtime overhead 可量化；
8. 阶段 2 baseline 已经冻结。

若第 8 条不满足，则输出：

```text
Stage3MechanismDecision = DEFERRED_BY_STAGE2
```

不得输出 `PASS`。

## 十六、阶段 3 通过标准

阶段 3 最终完成必须同时满足 implementation pass 与 mechanism pass。

### （二）mechanism pass

```text
Stage3MechanismDecision == PASS
```

1. 阶段 2 mechanism 已完成；

6. trajectory controls 不能复现主增益；

8. runtime overhead 有量化结果。

## 十七、阶段 3 不通过时的处理

若阶段 3 未通过，不应回滚阶段 3 代码接口。正确处理方式是：


若失败原因是 `stage2_mechanism_not_passed`，则不应解释为 trajectory 失败，而应解释为：

```text
阶段 3 mechanism audit 尚不具备正式判定条件。
```

若失败原因是 `trajectory_control_matches_main_gain`，则说明 trajectory statistic 可能没有提供密钥条件的独立证据。

若失败原因是 `trajectory_score_highly_correlated_with_tubelet`，则说明 `S_traj` 可能只是 tubelet evidence 的重复表达。
