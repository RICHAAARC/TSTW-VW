# 项目结构调整

## 一、文档定位

本文档是依据 `文件组织.md`、当前 `docs/` 治理约束以及当前仓库真实结构制定的项目结构调整方案。本文档不修改算法机制，不定义新的实验结论，不替代阶段 2 真实测试修复文档。其目标是将当前混合在 `main/` 中的核心方法代码、阶段性验证流程、Colab 执行流程、结果检查与打包工具重新划分边界，使项目在后续全生命周期内同时满足以下要求：

```text
1. main/ 仅保存最终发布版必须保留的核心方法、核心协议、核心评估、训练与命令行复现能力；
2. paper_workflow/ 仅保存 Colab notebook 与 Colab session wrapper，默认不进入最终发布版；
3. experiments/ 保存阶段性机制验证、论文实验协议、阶段 runner 与阶段 artifact builder；
4. scripts/ 保存数据准备、模型准备、结果检查、结果打包和 release 辅助工具；
5. docs/builds/、audit_reports/、.codex/ 作为内部构建与审计材料，不进入最终发布版；
6. 最终发布版通过命令行、配置文件、数据准备脚本与结果检查脚本复现，不依赖 Colab notebook。
```

本文档面向当前代码状态，优先解决以下混合问题：

```text
main/colab/ 位于核心包中；
main/protocol/real_video_vae_latent_runner.py 同时承担阶段 2 runner、Colab runtime config、artifact layout 与 formal gate 适配；
main/protocol/real_video_vae_latent_paths.py 固化 colab_* 输出路径；
main/analysis/real_video_vae_latent_artifacts.py 同时承担阶段 2 artifact builder、governance summary、formal checker 适配；
paper_workflow notebook 直接依赖 main.colab.*；
tests 中大量 Colab contract 测试直接依赖 main.colab.*；
placeholder / random / synthetic scaffold 文件仍分布在 main/ 和 configs/ 顶层。
```

## 二、调整总原则

### （一）方法代码与执行环境分离

`main/` 只表达项目最终方法“能做什么”，不表达“在 Colab 中如何执行”。因此，以下能力可在 `main/` 中存在：

```text
水印嵌入；
水印检测；
tubelet partition；
temporal synchronization；
evidence score；
fusion rule；
VAE backend；
真实视频 IO；
真实攻击；
统计阈值校准；
records / thresholds / manifest；
table rebuild；
质量与时序指标；
训练或微调；
命令行复现入口。
```

以下能力不得位于 `main/`：

```text
Google Drive 挂载；
Colab runtime preflight；
Colab session manifest；
Drive result packager；
notebook output checker wrapper；
notebook cell 顺序契约；
FAMILY_ID 目录管理；
Colab 本地 /content/TSTW_runtime/ 管理；
tar.zst 写回 Drive；
单阶段 Colab notebook 的运行状态检查。
```

### （二）核心协议与阶段验证分离

`main/protocol/` 应保存通用协议能力，例如 split、event plan、calibration、detection、evaluation、output layout。它不应长期保存某一阶段专用 runner。

当前的：

```text
main/protocol/real_video_vae_latent_runner.py
main/analysis/real_video_vae_latent_artifacts.py
```

属于阶段 2 机制验证流程，应迁入：

```text
experiments/real_video_vae_latent_probe/
```

迁移后，`experiments/real_video_vae_latent_probe/runner.py` 可以继续调用 `main/` 中的通用方法、视频、VAE、攻击、协议与分析模块，但 `main/` 不反向依赖 `experiments/`。

### （三）Notebook 与发布复现分离

`paper_workflow/` 是开发期 Colab workflow，不是最终发布版默认内容。最终发布版应提供命令行入口，例如：

```text
python -m main.cli.prepare_dataset ...
python -m main.cli.embed ...
python -m main.cli.detect ...
python -m main.cli.evaluate ...
python -m scripts.package_results ...
```

Colab notebook 可以作为内部执行入口继续存在，但 notebook 应调用 `experiments/` 或 `scripts/`，而不是要求核心 `main/` 暴露 `main.colab.*`。

### （四）迁移优先于删除

当前阶段 2 真实测试仍在修复中，不能一次性删除全部 scaffold。调整策略应为：

```text
先新增目标目录与兼容 wrapper；
再迁移 import；
再更新测试；
再收紧 audit；
最后删除旧路径或将旧路径标记为 deprecated。
```

不得在 P4 runner 闭环未完成前大规模删除当前阶段 2 可运行链路。

```text
4. packager 同时需要旧文件名与新文件名，出现重复或缺失；
5. harness 仍要求 placeholder config 位于顶层 configs/；
6. P4 阶段 2 真实测试修复与结构迁移同时进行，导致问题混淆。
```

### （二）回滚策略

每一批迁移都必须单独提交，并保留旧路径兼容一轮：

```text
commit A：新增目标目录与文档；
commit B：迁移 main/colab/ 到 wrapper；
commit C：替换 notebook import；
commit D：迁移 output layout 字段；
commit E：迁移 stage runner；
commit F：移除 deprecated wrapper。
```

如果测试失败，只回滚当前 commit，不回滚已经完成且通过测试的迁移。
