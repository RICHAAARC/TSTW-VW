## 一、阶段 2 的定位

阶段 2 应定义为 **real video VAE latent probe**，即“真实视频 VAE latent 机制验证”。其核心目标不是接入完整 DiT / Flow Matching 视频生成模型，也不是开始构建完整论文主表，而是在阶段 1 已经验证 `temporal-synchronized tubelet code` 的基础上，将受控 synthetic video latent 条件下成立的 tubelet / sync 机制迁移到真实视频经 VAE encode、decode 与 re-encode 后形成的 latent 空间中。

阶段 2 只回答一个核心机制问题：

```text
在真实视频经 VAE 编码、解码、攻击与重编码后，Tubelet+Sync 是否仍能在 fixed low-FPR protocol 下保持对时间攻击、空间攻击、压缩攻击和 local clip detection 的鲁棒性，同时保证视频质量与时序一致性损失可控。
```

因此，阶段 2 的最小结论应为：

```text
本阶段只证明：
真实视频可以被稳定读取、编码、解码、攻击和重编码；
frame_prc、tubelet_only、tubelet_sync 三类 variant 可以在真实视频 VAE latent 上复用同一协议运行；
Tubelet+Sync 在真实 VAE latent 中没有因 VAE encode-decode 噪声、压缩攻击或空间攻击而机制崩溃；
attacked negative FPR 仍由 calibration negative 控制；
视频质量损失、时序一致性损失与失败样例可以被 records 自动追溯；
Colab notebook 可以作为远程 GPU 执行入口，完成配置、代码拉取、Google Drive 挂载、模型加载、阶段 2 运行、结果检查与结果打包保存。
```

阶段 2 不应证明：

```text
Flow Matching trajectory statistic 具有独立增益；
DiT / Flow Matching sampling-time weak constraint 有效；
trajectory-aware embedding 可以提升最终鲁棒性；
完整论文主表、baseline comparison 与投稿级全量协议已经冻结；
真实视频生成模型端到端水印嵌入已经完成。
```

阶段 2 的正确工程含义是：在不破坏阶段 0 与阶段 1 协议层的前提下，将 `synthetic_video_latent` 替换或扩展为 `real_video_vae_latent`，将 synthetic latent attack 扩展为真实视频压缩、空间、模糊、噪声与时间攻击，并新增质量与时序一致性评估链路。

### （五）第五步：新增 `real_video_vae_latent.py`

新增：

```text

```

该模块负责把真实视频转为阶段 1 方法可消费的 latent tensor，并负责 VAE decode / re-encode 闭环。推荐流程如下：

```text
读取 source video
→ 标准化帧数、fps、分辨率
→ VAE encode 得到 clean latent z
→ 对 watermarked_positive 执行 tubelet embedding 得到 z'
→ VAE decode 得到 watermarked video x'
→ 对 clean_negative / watermarked_positive 施加 video attack
→ 对 attacked video 重新 VAE encode 得到 \hat{z}
→ 调用阶段 1 detector 计算 evidence_scores
```

```json

  "split": "calibration|test",
  "sample_role": "clean_negative|attacked_negative|watermarked_positive|attacked_positive",
  "video_source_id": "string",
```

## 十一、阶段 2 的最终判断口径

阶段 2 的最终判断不应写成“方法已经完整适用于 DiT / Flow Matching 视频生成模型”，而应写成：

```text
阶段 2 验证 temporal-synchronized tubelet code 在真实视频 VAE latent 条件下没有发生机制性崩溃。具体而言，真实视频经 VAE encode、decode、attack 与 re-encode 后，tubelet-level code 仍可提供跨帧承载证据，temporal synchronization 仍可在时间攻击与局部片段检测条件下提供对齐恢复能力，fixed low-FPR protocol 仍可通过 calibration negative 控制 clean 与 attacked negative 上的假阳性率。同时，阶段 2 建立了视频质量、时序一致性、失败样例导出与 Colab 远程 GPU 复现实验链路。
```

若阶段 2 通过，则可以进入 Flow Matching trajectory statistic probe。若阶段 2 不通过，则说明真实视频 VAE latent 中存在分布落差、VAE 编码噪声、压缩破坏、同步失败或质量损失问题，不应通过更复杂的 trajectory 或 sampling-time 模块掩盖问题。

## 当前阶段 2 机制修订: aligned payload safety gate

本节记录当前阶段 2 的实际实现口径, 用于覆盖早期文档中仍停留在宽搜索或 `sync_candidate_score` gate 的历史描述。

### 1. 阶段 2 当前目标

阶段 2 当前目标是完成 `real_video_vae_latent_probe` 条件下的机制证明收尾。证明重点不是扩大搜索空间, 而是在固定已验证 anchor 的前提下, 验证 `tubelet_sync` 是否能通过对齐后的 payload 证据提升 temporal crop 或 local clip 的检测能力, 同时保持 calibration negative 安全。

### 2. 默认 gate 规则

当前默认 rescue gate 为:

```text
sync_confidence_gate_rule = aligned_payload_safety_gate
```

该 gate 的正式字段为:

```text
S_payload_aligned
S_payload_rescue_gain
min_payload_rescue_gain
min_aligned_payload_score
sync_alignment_coverage_ratio
min_sync_alignment_coverage_ratio
sync_alignment_matched_count
min_sync_alignment_matched_count
```

其中 `sync_candidate_score` 不再作为默认 rescue gate, 也不再作为默认搜索维度。它仅保留为 alignment search 的排序分数和诊断 trace 字段。

### 3. S_final 更新规则

当前更保守地采用 payload-only rescue 公式:

```text
如果 aligned_payload_safety_gate 通过:
    S_final = S_final_before_rescue + S_payload_rescue_gain
否则:
    S_final = S_final_before_rescue
```

其中:

```text
S_final_before_rescue = S_payload_unaligned
S_final_after_rescue = S_final
rescue_gain_clipped = false
```

该规则的主要约束是: `sync_candidate_score` 只负责选择 offset / scale, 不直接贡献 `S_final`。未通过 payload safety gate 的 alignment 也不能进入 `S_final`。

### 4. selector split 协议

阶段 2 candidate selection 只能使用:

```text
dev
calibration
```

`test` split 只能用于最终报告和最终机制审计, 不能用于选择 `min_payload_rescue_gain`、`min_aligned_payload_score`、sync search radius 或 fusion rule。selector 输出必须显式记录:

```text
selector_split_policy = dev_calibration_only
test_split_used_for_selection = false
```

### 5. negative safety 统计

阶段 2 仍保留保守规则:

```text
negative_rescue_over_threshold_count = 0
```

同时需要记录样本量和置信上界, 避免把 `0 / 20` 与 `0 / 500` 混为同等强度的证据:

```text
calibration_negative_count
attacked_calibration_negative_count
negative_rescue_over_threshold_count
negative_rescue_over_threshold_rate
upper_confidence_bound_for_negative_rescue_rate
```

### 6. 当前默认搜索空间

当前默认 grid 使用固定 anchor 与小规模 gate 搜索:

```text
anchor = tubelet_only_cal_tl04_sp04x04_w009_em1000
lambda_sync = [0.01]
sync_search_radius = [8]
min_sync_positive_margin = [0.0]
min_sync_alignment_coverage_ratio = [0.125]
min_sync_alignment_matched_count = [64]
sync_confidence_gate_rule = [aligned_payload_safety_gate]
min_payload_rescue_gain = [0.01, 0.02]
min_aligned_payload_score = [0.095, 0.10]
min_sync_candidate_score = removed from default grid
```

该配置的目标是用尽可能少的候选验证机制闭环, 而不是继续进行宽搜索。
